# 運用手順

この文書は `discordbot-auto-reaction` を日常運用するための手順をまとめたものです。

## 1. 運用上の基本方針

この Bot は、チャンネル単位で自動リアクション設定を持ちます。

設定は MongoDB に永続化され、起動時にメモリキャッシュへ読み込まれます。通常投稿時は MongoDB を毎回参照せず、メモリキャッシュの内容に基づいてリアクションを付与します。

## 2. 日常的な操作

## 2.1 自動リアクションを設定する

対象チャンネルで、チャンネル管理権限を持つユーザーが次の形式で投稿します。

```text
@BotName 👍 ✅ 🎉
```

成功すると、Bot が `自動リアクション設定を追加しました` と返信します。

既存設定がある場合、新しい絵文字リストで上書きされます。追加ではなく置き換えです。

## 2.2 自動リアクション設定を解除する

対象チャンネルで Bot メンションのみを投稿します。

```text
@BotName
```

成功すると、Bot が `自動リアクション設定を解除しました` と返信します。

## 2.3 設定内容を確認する

対象チャンネルで次のコマンドを実行します。

```text
/リアクション設定確認
```

設定がある場合は Embed で表示されます。設定がない場合は `リアクション設定がありません` と表示されます。

## 2.4 チャンネル内の既存メッセージへ補完する

対象チャンネルで次のコマンドを実行します。

```text
/リアクション補完
```

この操作は `interaction.channel.history(limit=None)` によりチャンネル全履歴を走査します。
大規模チャンネルでは実行時間と API リクエスト数が大きくなるため、実行前に対象チャンネルの規模を確認してください。

## 2.5 単一メッセージへ補完する

補完したいメッセージを右クリックし、アプリメニューから `リアクション補完` を実行します。

チャンネル全体ではなく 1 件だけ補完したい場合に使用します。

## 2.6 自分のメッセージのリアクション内訳を確認する

自分が投稿したメッセージを右クリックし、アプリメニューから `リアクション確認` を実行します。

結果は DM に送信されます。他人のメッセージに対しては実行できません。

## 3. 権限管理

## 3.1 Bot 権限

対象チャンネルで次の権限が必要です。

- チャンネルを見る
- メッセージを送信
- メッセージ履歴を読む
- リアクションを追加
- 外部の絵文字を使用
- 埋め込みリンク
- アプリケーションコマンドを使用

権限が不足している場合、自動リアクションや補完が失敗します。リアクション付与失敗は現行実装ではユーザーへ詳細通知されません。

## 3.2 ユーザー権限

| 操作 | 必要条件 |
| --- | --- |
| 自動リアクション設定・解除 | チャンネル管理権限 |
| 設定確認 | 管理者 |
| チャンネル補完 | 管理者 |
| 単一メッセージ補完 | 管理者 |
| リアクション内訳確認 | 自分のメッセージのみ |

## 4. MongoDB 運用

## 4.1 使用データ

| 項目 | 値 |
| --- | --- |
| database | `auto_reaction` |
| collection | `emojis` |

ドキュメント例:

```json
{
  "channel_id": 123456789012345678,
  "emojis": ["👍", "✅"]
}
```

## 4.2 バックアップ

Bot の設定は MongoDB に保存されます。
バックアップ対象は `auto_reaction.emojis` コレクションです。

MongoDB Atlas を使用している場合は、Atlas のバックアップ機能またはエクスポート機能を使用してください。

## 4.3 DB 直接編集時の注意

Bot は起動時に DB 内容をメモリキャッシュへ読み込みます。
稼働中に MongoDB を直接編集しても、Bot のメモリキャッシュには即時反映されません。

DB を直接編集した場合は Bot を再起動してください。

## 5. 再起動手順

1. 稼働中プロセスを停止する
2. 必要に応じて環境変数を更新する
3. `python main.py` で再起動する
4. Bot がオンラインになることを確認する
5. `/リアクション設定確認` で既存設定が復元されていることを確認する

Railway などの PaaS では、環境変数の変更や再デプロイにより再起動されます。

## 6. リリース手順

推奨するリリース手順は次のとおりです。

1. main から作業ブランチを作成する
2. 変更内容を実装する
3. ローカルで `python main.py` の起動確認を行う
4. テスト用 Discord サーバーでコマンド同期と主要機能を確認する
5. Pull Request を作成する
6. レビュー後、main へマージする
7. 本番環境を再デプロイする
8. 本番サーバーで設定確認と自動リアクション付与を確認する

## 7. 動作確認チェックリスト

リリース後または環境変更後、次を確認してください。

- Bot がオンラインである
- Slash Command が表示される
- Message Context Menu が表示される
- `@BotName 絵文字...` で設定できる
- 設定後、新規投稿へ自動リアクションが付く
- `/リアクション設定確認` が正しく表示される
- `/リアクション補完` が実行できる
- 右クリック `リアクション補完` が実行できる
- 右クリック `リアクション確認` の結果が DM に届く
- Bot 再起動後も設定が残っている

## 8. チャンネル補完の運用注意

`/リアクション補完` はチャンネル全履歴を対象にします。

実行前に次を確認してください。

- 対象チャンネルのメッセージ数
- 設定されている絵文字数
- Bot に Add Reactions 権限があるか
- Bot に Read Message History 権限があるか
- カスタム絵文字が Bot から使用可能か

大規模チャンネルでは、まず単一メッセージ補完で設定と権限を確認してから実行することを推奨します。

## 9. カスタム絵文字の運用注意

カスタム絵文字を設定する場合、Bot がその絵文字を使用できる必要があります。

使用できないケースの例です。

- Bot が参加していないサーバーの絵文字
- 外部絵文字の使用権限がないチャンネル
- 削除済みの絵文字
- 絵文字 ID が無効

現行実装では、リアクション付与に失敗した絵文字は黙ってスキップされます。

## 10. DM 機能の運用注意

`リアクション確認` は結果を DM に送信します。

ユーザー側の設定により DM を受け取れない場合、結果は届きません。現行実装では DM 不達時の専用メッセージは定義されていません。

## 11. Secret 管理

次の値は公開リポジトリにコミットしないでください。

- Discord Bot Token
- MongoDB 接続 URL
- MongoDB ユーザー名
- MongoDB パスワード

`.env` は `.gitignore` に含まれていますが、`constants.py` に直接 Secret を書く運用は避けてください。

推奨は `.env` またはホスティングサービスの環境変数から読み込む運用です。

## 12. 監視項目

最低限、次の観点で監視してください。

- Bot プロセスが落ちていないか
- Discord への接続が維持されているか
- MongoDB 接続エラーが出ていないか
- コマンド同期時にエラーが出ていないか
- リアクション補完時に大量のエラーやレート制限が発生していないか

## 13. 障害時の切り分け順

1. Bot がオンラインか確認する
2. ホスティング環境のログを確認する
3. 環境変数 `TOKEN` / `MONGO_URL` を確認する
4. Discord Developer Portal の Intents を確認する
5. 対象チャンネルの Bot 権限を確認する
6. MongoDB 接続状態を確認する
7. `/リアクション設定確認` で設定が読めるか確認する
8. 単一メッセージ補完で最小動作を確認する
9. 必要に応じて Bot を再起動する
